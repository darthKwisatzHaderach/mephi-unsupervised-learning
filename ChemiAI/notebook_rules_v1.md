Вот подробная инструкция по оформлению Jupyter Notebook для передачи ИИ, основанная на лучших практиках из существующих гайдов. Инструкция содержит конкретные правила и примеры, чтобы ИИ мог следовать им неукоснительно. Вы можете скопировать этот текст и передать его ИИ.

## 🧑‍💻 ИНСТРУКЦИЯ ПО ОФОРМЛЕНИЮ JUPYTER NOTEBOOK ДЛЯ ИИ

Твоя задача — создавать Jupyter Notebook (`.ipynb`), следуя правилам, изложенным ниже. Эти правила делают ноутбуки читаемыми, воспроизводимыми и пригодными для совместной работы.

### 1. ОБЩИЕ ПРИНЦИПЫ

**Принцип 1.1: Celл — минимальная логическая единица.** Разбивай код и текст на **много маленьких ячеек** — по одному действию или идее на ячейку. Это упрощает редактирование, отладку и понимание.

**Принцип 1.2: Воспроизводимость превыше всего.** Ноутбук должен выполняться линейно сверху вниз. Перед фиксацией результата всегда используй **«Restart & Run All» (Kernel → Restart & Run All)** и исправляй ошибки, которые при этом возникают.

**Принцип 1.3: Никаких предположений.** Не думай, что читатель знает всё то же, что и ты. Определяй все неочевидные термины и сокращения при первом использовании. Избегай слов, которые могут смутить или принизить читателя («очевидно», «легко», «просто»).

### 2. СТРУКТУРА И СОДЕРЖАНИЕ

Следуй логической структуре — как в научной публикации:

**Ячейка 1: Markdown — Заголовок и метаданные.** Начинай каждый ноутбук с Markdown-ячейки, содержащей:
*   **Заголовок (уровень 1, `#`)** — только один на весь ноутбук.
*   **Автор(ы)** и **дата последнего изменения**.
*   **Аннотацию (Abstract):** краткое описание целей и ключевых выводов.

**Ячейка 2: Code — Все импорты и установка.** Помещай все `import` и команды `!pip install` (если нужны) в **первую исполняемую ячейку**.

**Последующие секции (Markdown Level 2, `##`, и ниже):**
*   **## 1. Введение и контекст:** Цели анализа, постановка задачи.
*   **## 2. Загрузка и понимание данных:** Источник данных, описание, датасет.
*   **## 3. Подготовка данных:** Очистка, трансформация, объяснение каждого шага.
*   **## 4. Анализ и моделирование:** Методология, код, визуализации.
*   **## 5. Интерпретация и обсуждение:** Результаты, ограничения, выводы.
*   **## 6. Заключение и дальнейшие шаги:** Резюме, что делать дальше.
*   **## 7. Приложение и ссылки:** Дополнительный код, источники данных и инструментов.

### 3. РАЗДЕЛЕНИЕ НА ЯЧЕЙКИ (Code vs Markdown)

**3.1. Секции.** Каждый заголовок раздела (`## Название`) должен начинаться с новой Markdown-ячейки, содержащей только этот заголовок.

**3.2. Параграфы.** В рамках одной Markdown-ячейки помещай **одно предложение на строку**. Это улучшает читаемость и контроль версий. **Не разрывай строки по символам** и не переноси вручную.

**3.3. Каждая задача — в своей Code-ячейке.** Помещай в одну Code-ячейку логически завершенный блок кода (например, загрузка данных, построение одного графика). Избегай создания ячеек, которые делают «все подряд».

### 4. MARKDOWN И ДОКУМЕНТАЦИЯ

**4.1. Текст и его форматирование.** Для текста вне заголовков используй обычный Markdown. Для выделения используй:
*   `*курсив*` или `**полужирный**`.
*   Для списков используй `-` или `1.`.
*   Для кода используй обратные кавычки ```.

**4.2. Изображения и таблицы.** Вставляй изображения через `![alt text](path.png)`. Для таблиц используй Markdown-синтаксис.

**4.3. Уравнения.** Для математики используй LaTeX: `$уравнение$` (в строке) или `$$уравнение$$` (отдельный блок).

**4.4. Стиль комментариев в коде.** Комментарии в Code-ячейках своди к минимуму. Если они необходимы (например, указать единицы измерения), делай их краткими и помещай в квадратные скобки: `# [m/s]`, `# [%]`. В идеале вся документация должна быть в Markdown-ячейках.

### 5. ФОРМАТИРОВАНИЕ КОДА И ЯЧЕЙКИ

**5.1. Длина строки.** Ограничивай длину строки кода **88 символами** (стандарт `black`). Это предотвращает появление горизонтальной прокрутки.

**5.2. Именование переменных и функций.** Используй понятные, описательные имена на английском (`user_age`, `calculate_mean`). Избегай общих названий вроде `f`, `x`, `data` и не переиспользуй имена для разных целей. Используйте `snake_case` для переменных и функций, `CamelCase` для классов.

**5.3. Размещение определений функций.** Давай **каждой функции свою собственную ячейку** и определяй её **только один раз** за ноутбук.

**5.4. Автоформатирование.** Используй автоформатирование (`black`, `yapf`). В JupyterLab установи расширение **`jupyterlab-code-formatter`**.

### 6. МАГИЧЕСКИЕ КОМАНДЫ (MAGIC COMMANDS)

Используй их для повышения эффективности. Различай **строчные (`%`)** и **ячеечные (`%%`)** магические команды.

*   **Измерение времени выполнения:**
    *   `%%time` — измеряет время выполнения всей ячейки.
    *   `%timeit` — усредняет время выполнения строки по нескольким запускам.
    *   `%%timeit` — то же для всей ячейки.
*   **Работа с файловой системой:**
    *   `%pwd`, `%ls`, `%cd`.
    *   `%%writefile filename.py` — сохраняет содержимое ячейки в файл.
    *   `%run script.py` — выполняет внешний скрипт.
*   **Отладка и окружение:**
    *   `%debug` — запускает отладчик после ошибки.
    *   `%who` / `%whos` — показывает переменные в текущем пространстве имен.
    *   `%reset` — очищает переменные.
    *   `%matplotlib inline` — для отображения графиков внутри ноутбука.
*   **Другое:**
    *   `%load_ext autoreload` и `%autoreload 2` — для автоматической перезагрузки модулей.
    *   `%load_ext` — загружает расширения.
    *   `%env` — показывает переменные окружения.

### 7. ОБЕСПЕЧЕНИЕ ВОСПРОИЗВОДИМОСТИ

**7.1. Замораживание зависимостей.** Всегда фиксируй точные версии всех используемых библиотек в файле `requirements.txt` или `environment.yml`. Используй инструменты вроде `pip-tools` или `Poetry`.

**7.2. Абсолютные пути и секреты.**
*   **Никогда не хардкодь абсолютные пути.** Используй относительные пути или переменные окружения.
*   **Никогда не храни секреты (API-ключи, пароли) в коде.** Используй переменные окружения и библиотеки вродe `python-dotenv`.

**7.3. Проверки (Assertions).** Добавляй `assert` для самопроверки критических состояний:
```python
assert df is not None, "DataFrame не загружен"
assert df.shape[0] > 0, "Датафрейм пуст"
assert not df['value'].isnull().any(), "Есть пропуски в колонке 'value'"
```
Это сломает выполнение при ошибке и укажет на её причину.

**7.4. Выполнение.** Перед любым экспортом/публикацией делай **«Kernel → Restart & Run All»**, чтобы убедиться в линейной воспроизводимости.

### 8. УПРАВЛЕНИЕ ВЕРСИЯМИ И СОВМЕСТНАЯ РАБОТА

**8.1. Чистые коммиты.** Никогда не коммить выходные данные ячеек (графики, результаты вычислений). Используй инструмент **`nbstripout`** для автоматической очистки ноутбуков от output'ов перед коммитом.

**8.2. Синхронизация с `.py`.** Для упрощения code review используй **`jupytext`** для автоматической синхронизации `.ipynb` с чистым `.py` файлом.

**8.3. Формат коммитов.** Названия коммитов должны быть на английском и следовать формату: `<type>: <description>`. Пример: `feat: add data validation for age column`.

### 9. ИНСТРУМЕНТЫ И АВТОМАТИЗАЦИЯ

**9.1. Статический анализ кода (Linting).** Используй `flake8` или `pylint` для проверки стиля кода.

**9.2. Pre-commit хуки.** Обязательно настрой **`pre-commit`** с хуками для форматирования и очистки ноутбуков. Пример конфигурации (`.pre-commit-config.yaml`) включает этапы: `nbstripout-preserve-timestamp` → `jupytext-enforce-pairing` → `jupytext-smart-sync`.

**9.3. Документация.** Для превращения ноутбуков в документацию используй **`nbsphinx`** (интеграция с Sphinx) или **Jupyter Book**.

### 10. СПЕЦИАЛЬНЫЕ ЗАМЕЧАНИЯ

*   **Относительные пути:** В любой обмен данными — только относительные пути (например, `../data/raw/file.csv`).
*   **Настройки ядра (kernel):** В production-ноутбуках после всех очисток **не сохраняй метаданные ядра** (`metadata.kernel_spec`).
*   **HTML:** Не используй HTML-теги внутри Markdown-ячеек без крайней необходимости — это снижает портируемость.
*   **Визуализации:** Делай их самодостаточными: добавляй подписи осей, легенды и заголовки. График должен быть понятен без кода, который его сгенерировал.
*   **Большие данные и время выполнения:** Если операция требует много времени или памяти, предупреди об этом читателя в Markdown-ячейке.
